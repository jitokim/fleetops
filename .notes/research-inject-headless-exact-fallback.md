# Research — inject headless-exact fallback

**기준일: 2026-07-17** (모든 근거는 이 날짜 기준 최신 확인)

## 문제 (유저 신호)

fleetops의 `i`(inject)는 P0-1/P0-2 액추에이션 가드에 의해, 한 디렉토리를 N개
세션이 공유하고 세션 레지스트리에 tty가 없으면 **무조건 거절**했다
(`refuseIfAmbiguous`). 캡틴이 실제로 orca 멀티세션 플릿(워크트리 1개에 세션
3개)을 운용하다 이 데드엔드에 부딪혔다 — orca CLI는 세션별 tty/tab 매핑
자체가 없어(2026-07-17 라이브 검증: `terminal list/show`에 tty 필드 없음)
Tier 1a로도 영원히 해소 불가. 거절 메시지는 "attach해서 직접 치라"였지만,
정작 같은 코드베이스의 429 auto-redrive는 **동일한 세션들을** Tier 2
(`claude --resume <정확한 SessionID> -p`)로 이미 무조건 헤드리스 구동하고
있었다. 즉 안전한(세션 고유, 오배송 불가) 경로가 이미 존재·신뢰되고
있는데도 인간의 inject만 막혀 있었다.

## 경쟁 신호

2026-07-17 오픈소스 경쟁 지형 리서치(awesome-agent-orchestrators,
awesome-cli-coding-agents, GitHub API 실시간 스타/활동 기준):

- 세션 매니저 레이어는 40+ 프로젝트 레드오션 (vibe-kanban 27.4k⭐,
  claude-squad 8.1k⭐, humanlayer 11.1k⭐ 등).
- fleetops의 차별점은 "control tower"(관찰+개입) 포지션과 **tty 기반 정밀
  액추에이션 + fail-closed 안전장치** — 경쟁작 대부분이 다루지 않는 영역.
- 경쟁작들은 멀티세션 공유 디렉토리 모호성을 아예 다루지 않거나(그냥 판에
  타이핑), 세션 단위 개입 자체가 없다.

이 기능은 그 차별점을 정확히 심화한다: "모호하면 거절"에서 "모호해도
**안전이 증명된 상태에서만** 세션 고유 경로로 배달"로 — fail-closed 원칙을
유지한 채(명시적 allowlist, Running은 여전히 거절) 커버리지를 넓힌다.

## 왜 지금인가 (1문단)

PR #35(2026-07-17 머지)가 이 기능의 전제 인프라를 방금 완성했다 —
TTYLocator 인터페이스, Tier 1a 일반화, 그리고 Tier 2 다운그레이드의 정직한
메시징. 그 직후가 데드엔드를 닫을 유일하게 올바른 타이밍이다: 인프라가
신선하게 검증돼 있고(전 게이트 green), orca 멀티세션 워크플로는 캡틴이
지금 매일 쓰는 실사용 경로이며, 미루면 "ambiguous = 거절"이라는 낡은
메시지가 신규 사용자의 첫인상으로 굳는다.

## 챌린저 1 (시니어 엔지니어) 통과 기록

- **"이 기능이 없어서 fleetops를 안 쓰는 사람이 실존하나?"** — 실존한다:
  캡틴 본인(현재 유일한 실사용자이자 디자인 파트너). orca 멀티세션 플릿에서
  idle 세션에 프롬프트를 넣으려면 fleetops를 떠나 수동으로
  `claude --resume <id> -p`를 쳐야 했다. 세션 ID를 눈으로 복사하는 순간
  cockpit의 존재 이유가 무너진다.
- **"기존 도구로 이미 되지 않나?"** — 수동 CLI로는 된다. 하지만 그 명령을
  안전하게 조립하는 데 필요한 정보(정확한 SessionID, 세션 상태)가 바로
  fleetops가 이미 가진 것이고, 수동 조립은 오타·오배송(형제 세션 ID 복사)
  위험이 있다. 도구의 요지가 "아는 것을 안전하게 실행"이다.
- **"창피할 만큼 작게 줄이면 뭐가 남나?"** — 남는 것이 정확히 이번 구현이다:
  새 라우팅 메커니즘 0개, 새 상태 0개. 기존 keypress 가드에 eligibility
  allowlist 함수 1개를 끼우고, 이미 신뢰되는 Tier 2 폴스루가 일하게
  둔다(+정직한 메시지). diff의 대부분은 테스트와 설계 주석이다.
